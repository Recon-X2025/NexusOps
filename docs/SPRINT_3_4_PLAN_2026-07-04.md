## Sprint 3 — Automation Loop

**Goal:** close the automation loop. The platform has world-class automation *primitives* (action runtime, BullMQ approvals, rules engine, copilot) but the connective tissue — trigger evaluation and outbound emission — is missing. Cluster 7 is ~60/100; these builds move it to ~75. (`PLATFORM_GAP_PLATFORM_2026-07-03.md` §10)

### 3.1 — Workflow trigger layer (#1 priority for the cluster) — HIGH effort
- **Gap:** `triggerType` enum (`scheduled`/`webhook`/`ticket_created`) is stored but **never polled or evaluated** (`routers/workflows.ts:107-172`). The only live trigger path is the ticket business-rules engine + manual canary. Assets/HR/finance events cannot start workflows.
- **Build:** a scheduler (BullMQ repeatable job, ~1-min cadence) that scans `workflows` for `triggerType='scheduled'` (evaluate schedule vs `lastRunAt`) and a lightweight in-process event bus that fans domain events to workflows matching `triggerType`. Both call the **existing** action dispatcher (`workflows/actions/runtime.ts:26-52`) — actions already fire real side effects, so no new action code.
- **Invariant:** idempotent per (workflow, trigger-instance); no double-fire on scheduler overlap.

### 3.2 — Outbound webhook dispatcher (#2) — MEDIUM effort
- **Gap:** `webhooks` + `webhookDeliveries` tables are fully modelled (events array, HMAC secret, `attempts`/`nextRetryAt`) at `schema/integrations.ts:96-116`, but **no code ever creates a delivery or POSTs a subscriber** — only a read-only list endpoint exists (`routers/integrations.ts:576-578`). Inbound webhooks are REAL; outbound is schema-only.
- **Build:** (a) emit a `webhookDeliveries` row when a subscribed event fires (workflow completion, entity state change); (b) a dispatcher worker that watches `status='pending'`/`nextRetryAt<=now()`, signs the payload with the stored HMAC secret, POSTs, and updates `status`/`attempts`/`nextRetryAt` with backoff. Schema needs no change.
- **Invariant:** at-least-once delivery with idempotency key; HMAC signature verifiable by the subscriber.

### 3.3 — Generalise business-rules engine beyond tickets (#3) — MEDIUM effort
- **Gap:** the rules engine is **ticket-only** (`services/business-rules-engine.ts:90-151`); the evaluator is already entity-agnostic but is only invoked from ticket create/update.
- **Build:** call the same engine from HR/asset/invoice/contract create/update hooks. Optionally widen the condition DSL (currently only `status_category_is`, `field_changed`) with AND/OR + numeric/date comparators — flagged as a cheaper follow-on (#9), can defer.

### 3.4 — Fire the ITSM loops (Top-10 #7) — MEDIUM/HIGH
- **Gap:** ITOM event correlation never evaluated, on-call escalation timers never fire, deploy→incident MTTR always null (`PLATFORM_GAP_ITSM` — Events S, On-call S, MTTR P). These are the same "stored-but-never-evaluated" pattern.
- **Build:** reuse the 3.1 scheduler/event-bus to fire correlation and escalation timers; compute MTTR from deploy↔incident linkage. **Scope note:** this is the largest single item; consider landing 3.1–3.3 first and treating ITSM loops as the back half of the sprint (or spilling to a 3.5 follow-on).

**Sprint 3 sizing (from `PLATFORM_GAP_INDEX` §6):** ~5–7 eng-weeks.

**Test posture:** each new loop needs a real-Postgres integration test proving the trigger fires exactly once and the side effect lands (mirror the SLA-sweep/approval BullMQ tests). Money-touching webhooks: assert no financial mutation on delivery.

## Sprint 4 — Revenue + Audit-Defense

**Goal:** turn captured data into revenue intelligence and audit defensibility. CRM ~45, IT Asset ~42. (`PLATFORM_GAP_CRM_2026-07-03.md`, `PLATFORM_GAP_ITASSET_2026-07-03.md`)

### 4.1 — CPQ tax/GST + per-line discount + PDF (CRM #1) — MED/HIGH
- **Gap:** quote `total = subtotal × (1 − discountPct/100)` **omits CGST/SGST/IGST entirely** (`deals.ts:356-357`); line items hardcode `discount: 0` (`deals.ts:371`); no PDF renderer. High India-correctness risk.
- **Build:** wire the existing `computeGST()` (`packages/payroll-math`) into quote calculation; add per-line discount to the `lineItems` schema; reuse the Legal cluster's contract-PDF engine for quote rendering. Optionally add a discount-approval gate (mirror the existing deal-win approval at `deals.ts:173-194`).

### 4.2 — SAM installed-vs-entitled reconciliation / ELP (IT Asset #1, Top-10 #6) — HIGH
- **Gap:** license records + on-the-fly utilization exist (`assets.ts:497-519`) but there is **no installed-vs-entitled comparison, no ELP, no discovery feed** (`assets.ts:488-617`, `schema/assets.ts:186-226`). This is the exact audit-defense risk SAM exists for (M365 true-ups run $300k–$500k).
- **Build:** add `installedCount`/`complianceStatus`/`elpSnapshot` to the license schema; a `softwareDiscoveryLogs` import table; device fingerprint fields (`serialNumber`/`macAddress`) as the reconciliation prerequisite; an ELP computation (entitled − installed → compliant/at-risk/over-deployed) with periodic snapshots.

### 4.3 — Computed account health score (CRM #2) — MEDIUM
- **Gap:** CSM metrics are literal placeholder zeros (`csm.ts:170,210-211`); account `healthScore` is a manually-written static field (`accounts.ts:46`).
- **Build:** compute health from real signals — renewal proximity (the renewals view is already real), open/breached case counts, resolution time — replacing the hardcoded zeros. Wire CSM SLA/CSAT to actual ticket data.

### 4.4 — Fix lossy lead→deal conversion + lead scoring (CRM #3, #4) — LOW then MEDIUM
- **Gap:** conversion drops account/contact FK and hardcodes 10% probability (`leads.ts:65-68`); `score` is a static field defaulting to 0 (`schema/crm.ts:166`).
- **Build:** carry account+contact+stage+probability through conversion (cheap bug-class fix, do first); then a fit+behavioural lead-scoring rule engine.

### 4.5 — Cheap enablers (do alongside) — LOW
- Asset↔contract **expiry alerting** using the 3.x scheduler (the link column already exists from Sprint 0.3); APM rationalization is deferred.
- Customer portal access-gating is the highest-effort CRM item (`schema/portal.ts`) — recommend **deferring to Sprint 5** unless a deal needs it.

**Sprint 4 sizing (from `PLATFORM_GAP_INDEX` §6 + Cluster digests):** ~5–7 eng-weeks (Tiers 1–2 ~11–16 if customer portal is pulled in).

## Sequencing recommendation

1. **Sprint 3, first half:** 3.1 trigger layer → 3.2 webhook dispatcher → 3.3 rules generalisation (the three structural loop-closers; they compound).
2. **Sprint 3, second half:** 3.4 ITSM loops (reuses 3.1's scheduler).
3. **Sprint 4, Tier 1 (audit + revenue unblock):** 4.2 SAM ELP, 4.1 CPQ GST.
4. **Sprint 4, Tier 2:** 4.3 account health, 4.4 lead conversion+scoring.
5. **Sprint 4, cheap wins:** 4.5 expiry alerting.
6. **Defer to Sprint 5:** customer portal, APM rationalization, PMO scheduling, condition-DSL richness, RAG for copilot.

**Alternative track (if the lens is "SMB revenue now"):** interleave the four `SMB_NEEDED_NOW_BACKLOG` deal-blockers — CSAT loop (cheapest; `surveys` router exists) + mobile approve/manager views first, then core connectors, then SAML SSO last (security-critical, needs auth-subsystem owner review). CSAT and connectors overlap naturally with Sprint 3's automation work (both ride the event/notification loop).
